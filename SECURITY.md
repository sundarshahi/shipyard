# Security Policy

Drydock ships security-by-default behavior into the projects it builds, and holds its own codebase to the same standard.

## Supported versions

The latest released version on `main` receives security fixes. Older versions are not maintained.

| Version | Supported |
|---|---|
| 6.0.x   | Yes |
| < 6.0   | No  |

## Reporting a vulnerability

Please report security issues **privately** — do not open a public issue for an exploitable vulnerability.

- Preferred: open a [GitHub Security Advisory](https://github.com/sundarshahi/shipyard/security/advisories/new) on the repository.
- Alternative: email **shahithakurisundar@gmail.com** with the details.

Include:

- Affected file(s) or component (e.g. a hook script, a skill, the plugin manifest).
- Reproduction steps or a proof of concept.
- Impact assessment (what an attacker could do).

We aim to acknowledge reports within a few business days and to coordinate a fix and disclosure timeline with you. We follow responsible-disclosure practices and will credit reporters who request it.

## Scope

In scope:

- The plugin manifest, skills, shared protocols, templates, and hook scripts in this repository.
- The `secret-guard.sh` hook and other PreToolUse/SessionStart enforcement logic.

Out of scope:

- Vulnerabilities in projects *generated by* Drydock (report those to the generated project's owner).
- Issues in Claude Code itself or third-party tools the pipeline invokes — report those upstream.

## Handling secrets

Drydock's `secret-guard` hook blocks writing/committing secret files and scans staged diffs for credentials. Never commit real secrets to this repository. If a secret is exposed, rotate it immediately and report it via the channels above.
